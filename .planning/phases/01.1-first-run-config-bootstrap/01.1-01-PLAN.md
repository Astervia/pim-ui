<objective>
Land the Rust-side foundation for first-run config bootstrap: a platform-correct
config-path resolver, a hard-coded D-16 sane-default TOML template, two new
Tauri commands (bootstrap_config + config_exists), and the sidecar crash-on-boot
detection branch (D-18) — but route the crash signal through the EXISTING
state-changed event by enriching DaemonConnection's RpcError. NO new Tauri
event channel is introduced. No UI in this plan; downstream plans (01.1-02..04)
consume what this plan ships.

Purpose: SETUP-01 backend (path detection), SETUP-02 backend (config
materialization + chained daemon_start), SETUP-03 backend (crash-within-500ms
detection via existing Phase 1 Sidecar::Terminated path, surfaced through the
existing daemon://state-changed event). Without this plan on disk the UI
cannot detect missing config, cannot write a default config, and cannot
honestly surface a crash-on-boot.

Output: Two new Rust modules, two new Tauri commands, one extended sidecar,
one new DaemonConnection method (report_crash_on_boot) that re-uses the
existing emit_state path. Rust unit tests cover path resolution + template
rendering + crash enrichment. invoke_handler registration in lib.rs is
updated; the existing 6 commands stay green.

ARCHITECTURAL NOTE — what changed from the earlier draft: the previous draft
proposed a NEW Tauri event `daemon://crash-on-boot` plus a new emit helper.
That would have forced a 3rd `listen()` call in `use-daemon-state.ts`,
breaking the cross-phase W1 invariant `grep -c 'listen(' src/hooks/use-daemon-state.ts == 2`.
Instead, since `DaemonStateChange { state, error?, hello?, status? }` already
exists and carries error payloads on every transition (Phase 1 Plan 01-01
SUMMARY: lines 12-13), we transport the crash-on-boot info as an enriched
RpcError on a `state: error` transition. Same UX, same wire info, W1 contract
preserved.
</objective>
