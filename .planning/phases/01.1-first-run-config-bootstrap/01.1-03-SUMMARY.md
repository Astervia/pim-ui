## D-13 Boot Sequence — Implementation Map

| Step | Owner | Behavior |
|------|-------|----------|
| 1 (UI: button-disable) | FirstRunScreen | `<fieldset disabled={isBooting}>` cascades to all form controls; the [Start pim] button shows `booting…` and gets `aria-label="Starting pim"` |
| 2 (bootstrapConfig) | useConfigBootstrap | `setStatus("writing_config")` → captures path via configExists probe → calls bootstrapConfig({nodeName, role}) → on throw, sets error `{step: "write_config", message: "Couldn't write config to {path}: {reason}"}` |
| 3 (requestPermission) | useConfigBootstrap | `setStatus("requesting_permission")` → calls useTunPermission().requestPermission() — `false` continues, even an unexpected throw continues per D-13 |
| 4 (startDaemon) | useConfigBootstrap | `setStatus("starting_daemon")` → calls startDaemon() → on throw, sets error `{step: "start_daemon", message: "Daemon didn't start. Check docs/SECURITY.md §2.1 or the bundled pim-daemon at src-tauri/binaries/."}` (verbatim D-13 step-4 copy) |
| 5 (onBootstrapComplete) | FirstRunScreen | `setStatus("complete")` triggers `useEffect([bootstrap.status])` → `completedRef`-guarded `onBootstrapComplete()` invocation → AppRoot flips `bootstrapped=true` → renders `<App />` (AppShell) |

**Inline-error copy origin:** the verbatim D-13 strings are formatted INSIDE useConfigBootstrap and stored on `bootstrap.error.message` ready for direct render. FirstRunScreen just renders `{bootstrap.error.message}` inside a `<div role="alert">` — no copy duplication, no string interpolation in the screen.

**Dedupe guard:** `useRef(false)` slot is the stable, render-invisible mechanism for "fire exactly once". React StrictMode double-mounts in dev, so without the guard `onBootstrapComplete` would fire twice and AppRoot would call `setBootstrapped(true)` twice (idempotent, but noisy). The ref check (`completedRef.current === false`) plus immediate set-on-fire guarantees single dispatch.

## Hostname / Platform Discovery Decision

Both hostname pre-fill and platform detection fall back to non-Tauri paths:

- **Hostname:** literal `"pim-node"` (UI-SPEC explicitly allows this — "if discovery fails, the user will see `pim-node` in the Input and can edit"). Sanitized through the same `sanitizeNodeName()` the live-discovery path would use, so the user always starts on a valid value.
- **Platform:** `navigator.userAgent` substring match → `"linux" | "macos" | "windows" | "unknown"`. Used purely to disable the gateway radio on macOS/Win per D-08 — Rust still owns whether `share_my_internet` actually functions on a given OS.

**Why no `@tauri-apps/plugin-os`?** Verified via `grep -E '"@tauri-apps/(api|plugin-os)"' package.json` and `ls node_modules/@tauri-apps/api/` — only `@tauri-apps/api ^2` is installed; the OS plugin is a separate package. Adding it would require:
1. `pnpm add @tauri-apps/plugin-os`
2. `cargo add tauri-plugin-os` in `src-tauri/Cargo.toml`
3. `.add(tauri_plugin_os::init())` in `lib.rs`
4. Capability permission entry in `src-tauri/capabilities/`

For a UI gating hint where Rust is the source of truth, this is overkill. A future plan can wire a Rust `get_hostname` Tauri command (per CONTEXT L131-132 "Claude's discretion") if real hostname pre-fill becomes a UX requirement. For v1, manual edit + sanitize is sufficient.
