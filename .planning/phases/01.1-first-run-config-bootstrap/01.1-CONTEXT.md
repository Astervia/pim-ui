# Phase 01.1: First-run config bootstrap — Context

**Gathered:** 2026-04-24
**Status:** Ready for planning
**Source:** `/gsd:discuss-phase 01.1 --auto` — recommended defaults selected for every gray area (see `01.1-DISCUSSION-LOG.md` for alternatives considered).

<domain>
## Phase Boundary

Phase 01.1 is a decimal-phase insertion that closes a product-level gap discovered during Phase 2 runtime verification: clicking `[ Start daemon ]` silently failed because `pim-daemon` requires `pim.toml` at a platform-default path before it can boot, and the product promise forbids asking the user to create that file by hand.

**Ships:**

1. A **first-run surface** (`src/screens/first-run.tsx`) — a single-screen form rendered when no `pim.toml` exists. Captures device name + role and exposes `[ Start pim ]` + `[ Customize… ]` (the latter grayed as a Phase-3 stub).
2. A **router fork** in `src/main.tsx` (or a new `AppRoot` component) — checks config presence once at boot; shows first-run OR AppShell accordingly. No in-app navigation between the two: once the config exists, first-run is gone forever (re-derivable by deleting the config file).
3. A **`bootstrap_config` Tauri command** — takes `{ nodeName, role }`, materializes a sane-default `pim.toml` at the platform user path, creates parent dirs if missing, returns the absolute path.
4. A **`config_exists` Tauri command** — stat-checks the platform path; returns `{ exists: boolean, path: string }`.
5. **Sidecar crash detection (≤500 ms)** — extends Phase 1's `Sidecar::spawn` so `CommandEvent::Terminated` arriving within 500 ms of `daemon_start` escalates to `DaemonState::error` with the daemon's stderr captured as the error payload, instead of just tracing a `warn!` and letting the UI sit silent.
6. An **`DAEMON ERROR` banner enrichment** in `LimitedModeBanner` — when the error payload references a config-file problem (text-match `failed to load config` or `NotFound (config)`), the banner surfaces the config path and a single-line hint "config path: {path}". No retry button (Phase 3 ships the raw TOML editor).

**Does NOT ship (belongs to other phases):**

- **`[ Customize… ]` behavior** — stub only. Actually opens the Settings tab = Phase 3 CONF-01..07. For Phase 01.1, the button is visibly grayed with "(Phase 3)" hint.
- Raw TOML editing / dry-run validation (Phase 3 — CONF-06/07).
- Per-section Settings forms (Phase 3 — CONF-01..05).
- Daemon-restart-on-config-save (Phase 3 — lives with the Settings save flow; Phase 01.1 only writes config ONCE at first-run).
- Onboarding wizard framing / "Run solo vs Add peer nearby" choice (Phase 4 — UX-01). Phase 01.1 boots the daemon and drops the user on the Dashboard; peer-pairing actions are already present in the Phase-2 Dashboard.
- Identity regeneration / export / key file management (Phase 5+ / BACKUP-01/02 — deferred).

<decisions>
## Implementation Decisions

### Router + boot-sequence

- **D-01:** Add an `AppRoot` component (`src/app-root.tsx`) that sits between `main.tsx` and `AppShell`. On mount, `AppRoot` issues `callDaemon`-like one-shot `invoke("config_exists")` (NOT via the Phase 1 `callDaemon` JSON-RPC wrapper — this is a Tauri command, not a daemon RPC). While the check is in flight (≤100 ms on local filesystem), render a centered `█ pim · booting…` splash using the existing brand primitives. On resolve: `exists === false` → render `<FirstRunScreen />`; `exists === true` → render `<AppShell />`. No state machine between the two — boot is one-shot; the user transitions from first-run to AppShell only by successfully clicking `[ Start pim ]` (which triggers a full remount of AppRoot via a `onBootstrapComplete()` callback that flips an in-memory flag).
- **D-02:** `TunPermissionProvider` stays where it is (at `main.tsx` root, wrapping AppRoot). This means TUN permission is reachable from BOTH first-run and AppShell — no need to duplicate provider. The `[ Start pim ]` click path calls `requestPermission()` → on grant, continues; on skip, writes config anyway but surfaces a `LIMITED MODE` banner inside AppShell with "permission skipped" copy (Phase 1 already renders this state).
- **D-03:** First-run skips the Sidebar entirely — there's no navigation to expose until the user has a mesh. The AppShell sidebar only appears after bootstrap succeeds.

### Platform config path (SETUP-01)

- **D-04:** Resolve the config path IDENTICALLY on Rust side, matching the kernel's `DEFAULT_CONFIG` rules AS CLOSELY AS POSSIBLE WITHOUT REQUIRING ROOT. Specifically:
  - **Linux:** `$XDG_CONFIG_HOME/pim/pim.toml` or `~/.config/pim/pim.toml` if `XDG_CONFIG_HOME` unset. (The kernel's `/etc/pim/pim.toml` system path requires root; not appropriate for a user-space desktop app. Phase 01.1 writes user-scope; future `pim-daemon --config <path>` invocation will be passed via Tauri externalBin args — see D-08.)
  - **macOS:** `~/Library/Application Support/pim/pim.toml` (standard macOS app-support convention).
  - **Windows:** `%APPDATA%\pim\pim.toml` (e.g. `C:\Users\{user}\AppData\Roaming\pim\pim.toml`).
- **D-05:** Share the path resolution between Rust and the UI via a `config_path` field on the `config_exists` result. The UI never hard-codes paths; Rust is the authority on where the file lives. Phase 1 already has `src-tauri/src/daemon/socket_path.rs` — add a sibling `config_path.rs` with the same pattern.
- **D-06:** Parent directory creation is `bootstrap_config`'s responsibility. If `~/.config/pim/` doesn't exist, create it with 0o700 permissions (owner-only — identity info lives here via `~/.pim/node.key` in a future phase). On failure (permission denied, read-only FS), return a structured error the UI can render honestly.

### First-run screen content (SETUP-02)

- **D-07:** Two form fields only — device name (text input, pre-filled from `navigator.userAgent`-derived hostname best-guess; or a Tauri `os::hostname()` command — prefer the Tauri route, Phase 1 likely already has similar) and role (radio group). No "network interface" picker, no "listen port" picker, no "MTU" — all those live behind `[ Customize… ]` (Phase 3 stub).
- **D-08:** Role radio options:
  1. **"Join the mesh"** — default, always selected. Description: "Just be a node on the mesh. Other devices can pair with you. Relays happen automatically when needed."
  2. **"Share my internet"** — gateway mode. On macOS/Windows, the radio input is `disabled` with `aria-disabled="true"` and sublabel "Gateway mode is Linux-only today. Your device can still join a mesh as a client or relay." (copy verbatim from Phase 2 UX-plan §3h — already voice-locked). On Linux, it's selectable.
- **D-09:** Primary action is `[ Start pim ]` — a bracketed monospace button (existing brand primitive from Phase 2). Secondary action is `[ Customize… ]` — grayed, `disabled`, with hover tooltip "(Phase 3)". No "Skip" / "Cancel" — there's nothing to cancel; the app can't function without a config. Closing the window closes the app (Tauri default).
- **D-10:** Below the form, a small footer line: `config will be written to ~/.config/pim/pim.toml · edit later in Settings` (path resolved via the `config_exists` result). Honest surfacing — never hide where state lives.
- **D-11:** Form validation (client-side):
  - Device name is required (min 1 char, max 64 chars, must match `^[A-Za-z0-9_\\-\\.]+$`). Pre-filled hostname that doesn't match (rare: hostname with spaces or Unicode) is sanitized on pre-fill (replace invalid chars with `-`, truncate to 64).
  - On invalid input, the button stays enabled; clicking it shows an inline error "Device name must be letters, numbers, `_`, `-`, or `.` (1-64 characters)" below the input. No toast, no shake — declarative.
- **D-12:** Keyboard: Tab/Shift-Tab cycles form controls in reading order (name input → role radio → Start button → Customize button). Enter on the name field submits (fires Start). Esc is a no-op (nothing to cancel).

### Bootstrap mechanics (SETUP-02 continuation)

- **D-13:** `[ Start pim ]` click path (strictly ordered — any step failure shows an honest error and aborts without partial side-effects):
  1. Disable button → show `booting…` substate on the button text.
  2. `invoke("bootstrap_config", { nodeName, role })` → returns `{ path: string }` on success.
  3. `requestPermission()` from `TunPermissionProvider` → if skipped, continue to step 4 anyway (daemon will start; Limited Mode banner surfaces inside AppShell later).
  4. `startDaemon()` from Phase 1's `rpc.ts` → resolves when daemon reaches `running` state OR errors out (Phase 1 contract).
  5. Transition AppRoot to AppShell (via `onBootstrapComplete()`).
  - If any step 2-4 fails, stay on FirstRunScreen, button re-enables, error renders inline above `[ Start pim ]`. Error copy matches the failing step:
    - step 2 fail: `Couldn't write config to {path}: {reason}` (with the OS error)
    - step 4 fail: `Daemon didn't start. Check docs/SECURITY.md §2.1 or the bundled pim-daemon at src-tauri/binaries/.`
- **D-14:** `bootstrap_config` writes using atomic-rename: write to `pim.toml.tmp`, fsync, rename to `pim.toml`. Prevents half-written config if the app crashes mid-write (extremely unlikely, but cheap insurance since the daemon hard-fails on malformed TOML per the kernel study).

### Default `pim.toml` template

- **D-15:** The TOML template is Rust-side (hardcoded string + `{node.name}` + `{roles}` interpolation), NOT JavaScript-side. Reason: the UI must not ship with protocol-specific knowledge; Rust owns the wire contract with the kernel. Template lives at `src-tauri/src/daemon/default_config.rs` and exports a single function `render_default_config(node_name: &str, role: Role) -> String`.
- **D-16:** Default values (derived from kernel-study §2.2 defaults, choosing the most privacy-preserving + safest starting state):
  ```toml
  [node]
  name = "{node_name}"

  [interface]
  name = "pim0"         # Linux daemon creates this; macOS daemon re-maps to utunN automatically
  mtu = 1400
  mesh_ip = "auto"      # daemon picks a CIDR it hasn't seen

  [transport]
  listen_port = 0       # 0 = OS-assigned, avoids port conflicts on dev machines
  tcp = true
  bluetooth = false     # off — BLE drains battery, surface opt-in in Phase 3 Discovery
  wifi_direct = false   # off — same

  [discovery]
  broadcast = true      # on — Aria expects nearby discovery to just work
  bluetooth = false
  wifi_direct = false
  auto_connect = false  # OFF — trust must be deliberate (TOFU), never silent

  [security]
  authorization_policy = "trust_on_first_use"
  require_encryption = true

  [roles]
  gateway = {gateway}   # true if role == "share_my_internet", false otherwise
  ```
  Note: exact section names and field names are guesses aligned with the kernel study; the planner/executor MUST validate against the actual `pim-daemon` binary by trial-parsing (spawn the daemon with this config, check it accepts). If fields diverge, adjust the template — the whole POINT of this phase is the daemon runs against the file we write.
- **D-17:** No `[[peers]]` section is written — peers are discovered at runtime or added later via Phase 3 PEER-02. Zero-peer state is a first-class Dashboard render (Phase 2 D-14).

### Sidecar crash detection (SETUP-03)

- **D-18:** Extend `src-tauri/src/daemon/sidecar.rs`:
  - After `spawn()` returns successfully, capture `tokio::time::Instant::now()` as `spawned_at`.
  - In the stdout/stderr forwarding loop, when `CommandEvent::Terminated(payload)` arrives, check `spawned_at.elapsed()`:
    - If `< 500 ms`: treat as crash-on-boot. Emit a new Tauri event `daemon://crash-on-boot` with payload `{ exit_code: i32 | null, signal: i32 | null, stderr_tail: String (last 2 KB of stderr), elapsed_ms: u64 }`.
    - If `>= 500 ms`: existing behavior (daemon ran fine then exited normally or externally). Phase 1 already handles this.
- **D-19:** On the UI side, `useDaemonState` (Phase 2 Plan 02-01 spine) gains a handler for `daemon://crash-on-boot`. It merges into `DaemonSnapshot.lastError` with shape `{ kind: "crash_on_boot", path: string, stderr_tail: string, elapsed_ms: number }` and transitions `DaemonState` to `error`. `LimitedModeBanner` already renders on `error` per Phase 1 Plan 03 — it just needs the enriched copy path (D-20).
- **D-20:** `LimitedModeBanner` Phase-1 copy is: `LIMITED MODE · pim daemon is stopped. Start it to join the mesh.` When `lastError.kind === "crash_on_boot"`, switch to: `DAEMON ERROR · pim-daemon exited in {elapsed_ms} ms during startup. Check config at {path}.` with an expandable one-line stderr excerpt (`·› {stderr_tail_first_line}`) below. Declarative, no exclamation, no emoji. No `[ Retry ]` button (Phase 3 Settings owns the re-bootstrap or re-edit flow).

### Skip / edge cases

- **D-21:** If the user closes the window during the `booting…` state (step 2-4 of D-13), Tauri's existing window-destroy hook already kills the sidecar (Phase 1 wires this in `lib.rs`). The partially-written config (if bootstrap succeeded but daemon didn't) REMAINS on disk — subsequent launches see it and go straight to AppShell + Limited Mode banner, which is honest (config exists, daemon isn't running). The user can delete the config manually to re-trigger first-run; Phase 3 Settings will add a "Regenerate from scratch" action.
- **D-22:** If `config_exists` itself fails (filesystem error, permission issue), treat as "config does not exist" and render the first-run surface. Log the error to tracing. `[ Start pim ]` will attempt to write and fail honestly, re-rendering the error.
- **D-23:** There is no migration path from a pre-existing malformed `pim.toml`. If config exists but is malformed, AppShell loads, daemon start fails, `DAEMON ERROR` banner surfaces (via D-19/D-20). User must edit manually (Phase 3 adds raw TOML editor — deferred) or delete the file to re-trigger first-run.
- **D-24:** Single-instance semantics: if the app is already running and user double-clicks the dock icon, Tauri routes focus to the existing window (Phase 1 behavior). Bootstrap can never run twice concurrently because it's triggered from a button inside AppRoot which is mounted once per process.

### Claude's Discretion

- Exact Tailwind class tuning for the first-run screen's form layout, provided it follows Phase 2's monospace aesthetic, uses `CliPanel` as the wrapper, uses the brand-token palette only, and has no gradients / `rounded-*` / exclamation marks.
- Whether to implement `hostname()` via `@tauri-apps/api/os::hostname()` or a new `get_hostname` Rust command — either is fine; prefer the existing Tauri API if available in the project's Tauri version.
- Whether `config_exists` and `bootstrap_config` share a Rust module or live separately — whichever is cleaner given Phase 1's existing `src-tauri/src/daemon/` structure.
- How to animate the `booting…` ellipsis on `[ Start pim ]` (static dots fine, animated dots fine, spinner fine — the brand aesthetic accepts any monospace-compatible pattern).
- Exact scrolling/overflow behavior at small window heights — Phase 1 window-size policy governs; nothing new here.
- Unit-test strategy: use the Phase 1 compile-only type-test pattern for Rust surface types; pure unit tests via existing harness for TOML rendering.

### Established Patterns (from Phase 1 + Phase 2)

- **Rust error model:** `anyhow::Result<T>` at the command layer, structured errors at the wire layer (matches kernel RPC error codes).
- **snake_case on the wire:** `bootstrap_config` takes `{ node_name, role }` in snake_case on the JS side, matches Tauri serde defaults on the Rust side.
- **W1 single-listener:** the UI adds a fan-out handler for `daemon://crash-on-boot` inside `useDaemonState`, not a new `listen(...)` call anywhere else. `grep -c 'listen(' src/lib/rpc.ts` must stay `0`; `grep -c 'listen(' src/hooks/use-daemon-state.ts` must stay `2` (unchanged — the fan-out target is extended, not the subscription count).
- **Honest copy, no exclamation:** all new strings in this phase must pass `! grep -q "!"` on string-literal content (the Phase 2 bang-free policy carries forward).
- **No gradients, no `rounded-*`, no literal palette classes:** brand rules remain absolute. shadcn primitives are brand-overridden per the Phase 2 pattern (sheet/select/scroll-area all got `rounded-none` overrides).
- **Atomic commits per task:** every GSD executor task ends in a single focused commit.

### Integration Points

- **`src/main.tsx`:** one-line wrap — replace `<AppShell />` with `<AppRoot />`. `TunPermissionProvider` stays wrapping.
- **`src-tauri/src/lib.rs` `tauri::Builder::default().invoke_handler(generate_handler!(...))`:** add `bootstrap_config` and `config_exists` to the list.
- **`src-tauri/src/daemon/sidecar.rs::spawn()`:** add `spawned_at` capture + `Terminated`-within-500ms branch + new event emit. Do NOT change the signature or the happy-path Ok-return timing (Phase 2 Plan 02-01 depends on Ok-return meaning "process was spawned").
- **`src/lib/rpc.ts`:** add `bootstrapConfig(args)` and `configExists()` typed wrappers. Add `DaemonEvents.crashOnBoot = "daemon://crash-on-boot"` constant. Do NOT add `listen(`.
- **`src/hooks/use-daemon-state.ts`:** register a handler on the global fan-out for `crash-on-boot` event that merges into `snapshot.lastError`.

### Creative options the architecture enables

- Because `TunPermissionProvider` is at the root, first-run can call `requestPermission()` before `daemon_start` without duplicating the modal. One provider, two callers.
- Because `LimitedModeBanner` already handles `DaemonState::error`, Phase 01.1's crash-on-boot path only needs to enrich the error payload — no new UI component required for the error state.
