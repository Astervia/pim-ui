Five-step happy path from click to Dashboard. Any step 2-4 failure stops the
flow, re-enables the button, surfaces inline error copy, **and leaves no
partial side-effects** per D-13.

window-destroy hook kills the sidecar (Phase 1 `lib.rs`). Per D-21, a
partially-written config (step 2 succeeded, step 4 failed) stays on disk;
on next launch, `config_exists === true` → AppShell loads → daemon_start
fails at boot → LimitedModeBanner crash-on-boot variant (C1) renders. No
new UI in Phase 01.1 handles this case — it flows through existing paths.

## Accessibility notes

### Keyboard navigation

**Tab order (forward, D-12):**

```
1. Device name Input (autoFocus on mount — first focus)
2. Role radio group (as a single tab stop per WAI-ARIA radiogroup pattern)
   - ArrowDown / ArrowRight: next selectable option
   - ArrowUp / ArrowLeft:   previous selectable option
   - Space / Enter:         selects focused option
   - Disabled options are skipped by arrow-keys on macOS/Windows
3. [ Start pim ] button
4. [ Customize… ] button is NOT in tab order (disabled attribute removes it)
```

**Shift-Tab reverses.** Tab from `[ Start pim ]` goes back to the role radio.

**Shortcut keys:**

- `Enter` inside the device name Input submits the form (fires `[ Start pim ]` click path, D-12). Implemented via `<form onSubmit>` — Enter in any text-type input inside a single-submit-button `<form>` triggers submit by default.
- `Esc` is a no-op on FirstRunScreen (D-12 — nothing to cancel). Inside the TunPermissionModal (step 3 of D-13), `Esc` triggers the existing Radix `onOpenChange(false)` path which resolves the permission promise with `false` — inherited from Phase 1 behavior, not changed.

### ARIA + semantic roles

- `<main aria-label="first-run setup">` on the outer `<section>` wrapping FirstRunScreen; AppRoot's splash variant uses `<main aria-label="starting pim" aria-busy="true">`
- `<CliPanel>` is a `<section>` per its existing `cli-panel.tsx` L23 — the title is NOT a heading level (intentional per Phase 2; the `NEW NODE` label is functionally a fieldset caption). A visually-hidden `<h1 className="sr-only">Set up this pim node</h1>` sits at the top of FirstRunScreen to give SR users a document outline.
- Device name Input: `<label htmlFor="device-name">` explicitly associated; `aria-describedby="device-name-helper device-name-error"` (space-separated, only the present IDs are announced); `aria-invalid="true"` when `validationError` is set.
- Role radio group: `<fieldset>` + `<legend>` native semantics — announced as "role, radio group, 2 options". Each option is `<input type="radio">` + visible `<label>` pair. The custom `◆`/`○` glyph has `aria-hidden="true"` and the radio's native checked state carries the semantic.
- Disabled gateway row (macOS/Win): `aria-disabled="true"` on both the `<input>` and the `<label>`; additionally `tabindex="-1"` on the input to remove it from arrow-key traversal within the radiogroup (Radix-compatible).
- Validation error `<p role="alert">` — live region for immediate SR announce.
- Bootstrap-failure inline error (step 2 / step 4): `<div role="alert">` — same live-region treatment.
- `[ Start pim ]`: `<button type="submit" aria-disabled={booting ? "true" : undefined}>`; during booting, `aria-label="Starting pim, please wait"` replaces the visible text to avoid reading "booting" as user-actionable.

  *(Voice audit note: "please wait" breaks the STYLE.md §Always rule. Revise to `aria-label="Starting pim"` only. The omitted "please wait" keeps voice clean.)*

- `[ Customize… ]`: `aria-disabled="true"`; the native `title="(Phase 3)"` is read as the accessible description.
- TunPermissionModal (mid-flow step 3): unchanged from Phase 1 — Radix Dialog provides `role="dialog" aria-modal="true"` and focus trap.

### Focus-visible on the green-phosphor palette

- Default `:focus-visible` from `src/globals.css` L240 = `outline: 1px solid var(--color-ring)` + `outline-offset: 2px` — works but small.
- Phase 01.1 upgrade (consistent with Phase 2): large click targets (Input wrapper, role-radio rows, `[ Start pim ]` at `size="lg"`) use `outline-2 outline-ring outline-offset-2`. Applied per-component via className — **not** a globals.css override.
- Never use a focus outline `color` other than `--color-ring`.

### Color-alone conveys no meaning

- Role selected state: `◆` vs `○` glyph change **plus** `text-primary` vs `text-muted-foreground` color change **plus** native radio `checked` attribute carrying the semantic. Colorblind + SR + sighted paths all receive distinct signal.
- Disabled gateway row: `opacity-60` (hue-neutral dim) **plus** `cursor-not-allowed` **plus** `aria-disabled` **plus** the sublabel text explicitly stating "Gateway mode is Linux-only today." Four independent channels.
- Validation error: `text-destructive` color **plus** `role="alert"` live-region announce **plus** `aria-invalid` on the input **plus** explicit text stating the rule.

### Motion & reduced-motion

- Phase 01.1 introduces NO new animations beyond the existing `.logo-hero` block-cursor-blink + word-typing-reveal in `src/globals.css` L171-196.
- `prefers-reduced-motion` is honored globally (L199-210) — the splash + FirstRunScreen hero render statically under reduced-motion.
- Transition between splash → FirstRunScreen is a React state transition (no CSS transition, no animation). Fast, instant, reduced-motion-safe by default.
- Between FirstRunScreen → AppShell (D-13 step 5): instant React remount of AppRoot. No transition.
